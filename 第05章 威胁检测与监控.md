# 第5章 威胁检测与监控

## 5.1 威胁检测的技术体系

### 5.1.1 终端检测与响应（EDR）

EDR是终端安全的核心技术，提供终端层面的威胁检测和响应能力。

**EDR核心功能：**
- **行为监控**：监控终端上的进程、文件、网络活动
- **威胁检测**：识别恶意软件、异常行为
- **响应能力**：隔离、清除、取证分析

**EDR架构：**
```
终端Agent → 数据收集 → 云端分析 → 响应处置
```

**主流EDR产品：**
- CrowdStrike Falcon
- SentinelOne
- Microsoft Defender for Endpoint

### 5.1.2 网络检测与响应（NDR）

NDR通过分析网络流量来检测威胁。

**NDR核心功能：**
- **流量分析**：深度包检测和分析
- **异常检测**：识别异常网络行为
- **威胁狩猎**：主动搜索潜在威胁

**NDR工作原理：**
1. 收集网络流量数据
2. 建立正常行为基线
3. 检测偏离基线的异常行为
4. 关联威胁情报

### 5.1.3 安全信息与事件管理（SIEM/SOAR）

SIEM提供安全事件的集中管理和分析能力。

**SIEM核心功能：**
- **日志收集**：从多个来源收集日志
- **事件关联**：关联不同来源的安全事件
- **告警管理**：统一管理和响应告警

**SOAR扩展能力：**
- **自动化响应**：根据预设剧本自动响应
- **编排工作流**：设计复杂的响应流程
- **案例管理**：跟踪安全事件的处理过程

## 5.2 基于威胁情报的主动检测

### 5.2.1 威胁情报的类型

威胁情报分为不同级别：

| 级别 | 描述 | 例子 |
|------|------|------|
| **战术情报** | 具体攻击指标 | IOC（IP、域名、哈希） |
| **操作情报** | 攻击手法和技术 | TTP（技术、战术、程序） |
| **战略情报** | 威胁组织和动机 | APT组织分析 |

### 5.2.2 威胁情报的应用

**情报应用场景：**
- **实时阻断**：根据IOC阻断恶意流量
- **威胁检测**：利用TTP检测攻击行为
- **态势感知**：了解威胁格局和趋势

**情报整合流程：**
1. **情报收集**：从多个渠道获取情报
2. **情报分析**：验证和评估情报质量
3. **情报应用**：将情报导入检测系统
4. **效果评估**：评估情报的实际效果

## 5.3 基于ATT&CK框架的威胁狩猎

### 5.3.1 ATT&CK框架简介

MITRE ATT&CK是一个攻击战术和技术的分类框架。

**ATT&CK矩阵：**
- **战术**：攻击者的目标（如初始访问、执行、持久化）
- **技术**：实现战术的具体方法
- **子技术**：技术的具体变体

### 5.3.2 威胁狩猎流程

威胁狩猎是主动搜索潜在威胁的过程：

**狩猎流程：**
1. **假设生成**：基于威胁情报提出假设
2. **数据收集**：收集相关日志和流量数据
3. **分析验证**：验证假设是否成立
4. **响应处置**：对发现的威胁进行处置
5. **知识沉淀**：将发现反馈到检测规则

## 5.4 告警运营：从海量告警到有效事件

### 5.4.1 告警分诊与优先级判定

面对海量告警，需要进行有效的分诊：

**分诊流程：**
1. **初步筛选**：过滤误报和低优先级告警
2. **优先级判定**：根据风险评估确定优先级
3. **告警聚合**：将相关告警聚合为事件

**优先级判定因素：**
- 资产重要性
- 威胁等级
- 攻击阶段

### 5.4.2 告警富化与上下文关联

告警富化可以提供更完整的上下文信息：

**富化信息来源：**
- 资产信息
- 用户信息
- 威胁情报
- 历史事件

**上下文关联价值：**
- 理解攻击的完整链条
- 评估攻击的真实影响
- 支持准确的响应决策

### 5.4.3 实战案例：从告警海洋中打捞真实威胁的艰辛历程

**背景：** 某金融科技公司部署了一套商业SIEM，内置了超过500条关联规则。安全分析师每天面对超过2000条告警，疲于处理大量误报。某次真实攻击中，攻击者通过弱口令登录VPN后横向移动至核心数据库，整个过程产生了5条高危告警，但均被分析师当作"常规噪音"忽略。

**痛点与难点：**

- **痛点1：告警疲劳症候群**。每日海量告警中真实威胁占比不足1%，分析师产生"狼来了"心理。

- **痛点2：规则缺乏上下文调优**。规则未考虑资产重要性（比如普通测试服务器的暴力破解不应与核心数据库同等对待）。

- **痛点3：告警之间缺乏关联**。5条告警分别出现在不同的控制台界面，分析师无法将它们串联成一个攻击链。

**临时解决办法（当时怎么做）：**

- **停用告警"休克疗法"**：安全负责人做了一个激进决定——临时停用所有低级别告警规则，仅保留最核心的5条规则（如域控登录异常、敏感数据批量下载）。告警量从2000+骤降至20以内。

- **引入资产权重系数**：在SIEM中手动添加一个"资产重要性"字段（1-5分），让每条告警乘以权重系数后重新排序。至少让真正重要的告警能排在队列前面。

- **人肉关联分析**：指派一名资深分析师，专门负责每小时浏览一遍所有告警日志，用Excel进行人工关联（效率极低但有效）。

**落地建议（现在怎么做更好）：**

- 引入SOAR进行告警富化（如自动查询IP的地理位置、威胁情报、资产信息），帮助分析师快速判断。

- 使用ATT&CK框架对告警进行战术阶段映射，自动化构建攻击故事线。

- 探索基于UEBA的行为异常检测，减少对静态规则的依赖。

### 5.5.4 Sigma规则与ATT&CK战术映射实战

**Sigma规则示例：横向移动检测**

```yaml
title: Suspicious SMB Connection - Possible Lateral Movement
logsource: windows
detection:
  selection:
    EventID: 5140
    ShareName|contains: 'C$'
    AccessMask: '0x2'
  filter:
    IpAddress|startswith:
      - '10.'
      - '192.168.'
      - '172.16.'
  condition: selection and not filter
falsepositives:
  - Legitimate administrative access
level: High
tags:
  - ATT&CK.T1021.002
```

**Sigma规则转换为SPL查询：**

```spl
index=security EventCode=5140 
| search ShareName=*C$* AccessMask=0x2 
| search NOT IpAddress=10.* NOT IpAddress=192.168.* NOT IpAddress=172.16.* 
| stats count BY src_ip, dest_ip, ShareName 
| eval severity=if(count>5, "HIGH", "MEDIUM") 
| table _time, src_ip, dest_ip, ShareName, count, severity
```

**KQL多源关联查询示例：**

```kusto
// 横向移动攻击链分析
SecurityEvent
| where EventID == 4688 and ProcessName contains "psexec.exe"
| join kind=inner (
    DeviceNetworkEvents
    | where ActionType == "ConnectionSuccess"
) on DeviceName
| where TimeGenerated between (ago(1h)..now())
| summarize ConnectionCount = count() by SourceIpAddress, DestinationIpAddress
| where ConnectionCount > 3
| project SourceIpAddress, DestinationIpAddress, ConnectionCount
```

## 5.6 【AI深度进阶】AI驱动的智能检测体系

### 5.6.1 从规则引擎到AI模型：检测范式的根本变革

传统检测依赖规则引擎，存在局限性：

| 维度 | 规则引擎 | AI模型 |
|------|---------|--------|
| **检测能力** | 已知威胁 | 未知威胁 |
| **适应性** | 静态规则 | 动态学习 |
| **误报率** | 较高 | 较低 |
| **扩展性** | 有限 | 强 |

### 5.6.2 基于机器学习的异常行为检测与用户行为分析

**机器学习检测方法：**
- **监督学习**：使用标注数据训练模型
- **无监督学习**：发现未知模式
- **半监督学习**：结合标注和未标注数据

**用户行为分析（UBA）：**
- 建立用户正常行为基线
- 检测异常行为
- 识别内部威胁

### 5.6.3 Transformer模型在零日恶意软件检测中的应用

Transformer模型在恶意软件检测中展现出强大能力：

**应用场景：**
- **恶意代码检测**：分析代码特征
- **恶意文档检测**：分析文档内容
- **网络流量分析**：识别恶意流量模式

**技术优势：**
- 处理长序列数据
- 捕捉上下文关系
- 迁移学习能力

### 5.6.4 告警分诊智能体：自动化降噪与优先级判定

**告警分诊智能体功能：**
- **智能降噪**：过滤误报和低价值告警

- **告警聚合**：将相关告警合并为事件
