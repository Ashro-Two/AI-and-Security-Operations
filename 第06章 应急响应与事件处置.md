# 第6章 应急响应与事件处置

## 6.1 应急响应的生命周期（PICERL模型）

应急响应是一个系统化的过程，通常遵循PICERL模型：

| 阶段    | 名称                  | 核心活动           |
| ----- | ------------------- | -------------- |
| **P** | 准备（Preparation）     | 制定预案、建立团队、培训演练 |
| **I** | 识别（Identification）  | 发现和确认安全事件      |
| **C** | 遏制（Containment）     | 阻止事件扩散         |
| **E** | 根除（Eradication）     | 清除威胁根源         |
| **R** | 恢复（Recovery）        | 恢复系统正常运行       |
| **L** | 复盘（Lessons Learned） | 总结经验教训         |

## 6.2 应急响应预案设计与演练

### 6.2.1 预案设计原则

**预案设计应遵循以下原则：**

- **完整性**：覆盖各类安全事件场景
- **可操作性**：步骤明确、责任清晰
- **灵活性**：适应不同规模和类型的事件
- **可更新**：定期审查和更新

### 6.2.2 桌面推演 vs 实战演练

**桌面推演：**

- **形式**：模拟讨论，不实际操作
- **优点**：成本低、效率高
- **适用场景**：验证预案逻辑、团队协作

**实战演练：**

- **形式**：实际操作，模拟真实攻击
- **优点**：真实感强、检验实际能力
- **适用场景**：检验技术能力、响应流程

### 6.2.3 常见场景预案

**勒索软件响应预案：**

1. **识别阶段**：检测勒索软件活动
2. **遏制阶段**：隔离受感染终端，阻止横向移动
3. **根除阶段**：清除恶意软件，恢复数据
4. **恢复阶段**：验证系统完整性，恢复业务
5. **复盘阶段**：分析攻击入口，改进防护措施

**数据泄露响应预案：**

1. **识别阶段**：确认数据泄露范围和类型
2. **遏制阶段**：停止数据流出，保护敏感数据
3. **根除阶段**：修复漏洞，加强访问控制
4. **恢复阶段**：恢复正常业务，通知相关方
5. **复盘阶段**：评估影响，完善数据保护措施

**APT入侵响应预案：**

1. **识别阶段**：发现潜伏的APT活动
2. **遏制阶段**：限制攻击者活动范围
3. **根除阶段**：彻底清除攻击者植入的工具
4. **恢复阶段**：重建系统，加强监控
5. **复盘阶段**：分析攻击手法，更新检测规则

## 6.3 取证分析与根因溯源

### 6.3.1 取证分析流程

**取证分析是确定攻击根源和影响的关键步骤：**

**取证流程：**

1. **证据收集**：收集日志、内存、磁盘等证据
2. **证据保全**：确保证据完整性和可追溯性
3. **证据分析**：分析证据，重建攻击链
4. **报告编写**：提交取证报告

**取证类型：**

- **网络取证**：分析网络流量
- **主机取证**：分析终端系统
- **云取证**：分析云环境日志
- **移动取证**：分析移动设备

### 6.3.2 根因溯源方法

**根因分析技术：**

- **攻击链重建**：还原攻击的完整过程
- **时间线分析**：梳理事件发生顺序
- **关联分析**：关联不同来源的证据
- **威胁情报比对**：与已知威胁进行比对

## 6.4 事件复盘与持续改进

### 6.4.1 如何撰写高质量的事件复盘报告

**复盘报告应包含以下内容：**

- **事件概述**：事件发生时间、影响范围
- **攻击分析**：攻击手法、入口点、传播路径
- **响应过程**：采取的措施和效果
- **根因分析**：导致事件发生的根本原因
- **改进建议**：针对根因提出改进措施
- **行动计划**：明确改进措施的实施计划

### 6.4.2 从事件到整改：驱动安全建设的闭环

**复盘驱动改进的流程：**

```
事件发生 → 复盘分析 → 根因识别 → 整改措施 → 验证效果 → 持续优化
```

**关键成功因素：**

- 高层支持和资源投入
- 跨部门协作
- 持续跟踪和验证

### 6.5.4 从事件复盘到AI规则自进化

### 6.5.5 SOAR剧本开发实战

**剧本开发流程：**

```
需求分析 → 流程设计 → 剧本编写 → 测试验证 → 上线部署
    ↓           ↓            ↓           ↓           ↓
 响应场景   工作流设计   编写代码/拖拽   模拟测试   持续监控
```

**实战示例：恶意软件响应剧本**

**场景描述：** 检测到终端感染恶意软件后的自动响应

**剧本流程：**

```
1. 检测到恶意软件告警
   ↓
2. 收集终端信息（IP、主机名、用户）
   ↓
3. 查询终端是否在关键资产清单中
   ↓
4. 如果是关键资产 → 立即隔离
   ↓
5. 如果是非关键资产 → 通知用户并隔离
   ↓
6. 收集取证数据（进程、网络连接、日志）
   ↓
7. 通知安全团队
   ↓
8. 更新资产状态
```

**剧本代码示例（伪代码）：**

```python
def handle_malware_alert(alert):
    # 1. 收集终端信息
    endpoint = get_endpoint_info(alert['endpoint_id'])
    
    # 2. 查询资产重要性
    asset_importance = get_asset_importance(endpoint['asset_id'])
    
    # 3. 执行隔离
    if asset_importance == 'critical':
        isolate_endpoint(endpoint['ip'], immediate=True)
    else:
        notify_user(endpoint['user'])
        isolate_endpoint(endpoint['ip'], immediate=False)
    
    # 4. 收集取证数据
    collect_forensics(endpoint['ip'])
    
    # 5. 通知安全团队
    notify_security_team(alert, endpoint)
    
    # 6. 更新资产状态
    update_asset_status(endpoint['asset_id'], 'quarantined')
```

**剧本设计原则：**

| 原则       | 描述           | 示例         |
| -------- | ------------ | ---------- |
| **最小影响** | 响应措施应最小化业务影响 | 非关键资产延迟隔离  |
| **可回滚**  | 响应操作应可撤销     | 隔离后可恢复网络   |
| **可审计**  | 所有操作应记录日志    | 记录隔离时间和操作人 |
| **可配置**  | 剧本参数应可配置     | 资产重要性阈值可调整 |

**剧本优化技巧：**

- 添加条件分支，根据不同情况执行不同操作
- 集成威胁情报，增强决策能力
- 添加异常处理，确保剧本稳定性
- 定期审查和更新剧本

### 6.5.6 SOAR剧本代码示例

**StackStorm Playbook（YAML格式）：**

```yaml
name: ransomware_response
description: Automated response to ransomware detection
pack: security
runner_type: orquesta
entry_point: workflows/ransomware_response.yaml

workflow:
  version: 1.0
  description: Ransomware containment workflow
  input:
    - endpoint_id
    - severity: "HIGH"
  
  tasks:
    isolate_endpoint:
      action: network.isolate
      input:
        endpoint_id: "{{ ctx().endpoint_id }}"
        method: "full"
      next:
        - when: "{{ succeeded() }}"
          do: collect_forensics
    
    collect_forensics:
      action: endpoint.collect_evidence
      input:
        endpoint_id: "{{ ctx().endpoint_id }}"
        artifacts:
          - memory_dump
          - process_list
          - network_logs
      next:
        - when: "{{ succeeded() }}"
          do: notify_security_team
    
    notify_security_team:
      action: slack.post_message
      input:
        channel: "#security-alerts"
        message: |
          Ransomware detected!
          Endpoint: {{ ctx().endpoint_id }}
          Severity: {{ ctx().severity }}
          Action: Isolated and collecting forensics
```

**Python脚本实现资产查询与防火墙封禁：**

```python
import requests
import json

class SOARConnector:
    def __init__(self, api_key):
        self.api_key = api_key
        self.cmdb_url = "https://cmdb.example.com/api/v1"
        self.firewall_url = "https://firewall.example.com/api/v1"
    
    def query_asset(self, ip_address):
        """查询资产信息"""
        headers = {"Authorization": f"Bearer {self.api_key}"}
        response = requests.get(
            f"{self.cmdb_url}/assets?ip={ip_address}",
            headers=headers
        )
        return response.json()["results"][0]
    
    def block_ip(self, ip_address, reason):
        """在防火墙封禁IP"""
        headers = {"Authorization": f"Bearer {self.api_key}"}
        payload = {
            "ip": ip_address,
            "duration": 3600,
            "reason": reason
        }
        response = requests.post(
            f"{self.firewall_url}/block",
            headers=headers,
            json=payload
        )
        return response.json()
    
    def create_ticket(self, asset_info, reason):
        """创建工单"""
        headers = {"Authorization": f"Bearer {self.api_key}"}
        payload = {
            "title": f"Security Alert: {reason}",
            "description": f"IP: {asset_info['ip']}\nHostname: {asset_info['hostname']}",
            "priority": "HIGH"
        }
        requests.post("https://jira.example.com/api/v1/issues", headers=headers, json=payload)

# 使用示例
connector = SOARConnector(api_key="your-api-key")
asset = connector.query_asset("192.168.1.100")
connector.block_ip("192.168.1.100", "Ransomware detected")
connector.create_ticket(asset, "Ransomware detection")
```

### 6.5.7 安全运营自动化阶梯：从脚本到AI智能体

**自动化阶梯模型：**

| 阶段          | 自动化程度       | 特征     | 技术手段           | 适用场景    |
| ----------- | ----------- | ------ | -------------- | ------- |
| **Level 1** | 脚本自动化       | 手动编写脚本 | Shell/Python脚本 | 简单重复性任务 |
| **Level 2** | Playbook自动化 | 预设响应剧本 | SOAR平台         | 标准化响应流程 |
| **Level 3** | AI增强自动化     | AI辅助决策 | AI推荐+人工确认      | 复杂决策场景  |
| **Level 4** | AI智能体自动化    | 自主决策执行 | AI智能体          | 高确定性场景  |

**各阶段能力对比：**

| 能力       | Level 1 | Level 2 | Level 3 | Level 4 |
| -------- | ------- | ------- | ------- | ------- |
| **灵活性**  | 低       | 中等      | 高       | 高       |
| **决策能力** | 无       | 预设规则    | AI辅助    | AI自主    |
| **适应能力** | 差       | 有限      | 良好      | 优秀      |
| **人工干预** | 100%    | 部分      | 关键节点    | 监督      |
| **规模化**  | 有限      | 中等      | 高       | 无限      |

**进阶路径：**

```
脚本自动化 → Playbook自动化 → AI增强自动化 → AI智能体自动化
     ↓                ↓                ↓                ↓
   基础能力         标准化          智能化          自主化
```

**实践建议：**

1. **从简单开始**：先自动化重复性高、风险低的任务
2. **逐步升级**：随着经验积累，逐步提升自动化级别
3. **保持可控**：关键操作保留人工复核机制
4. **持续优化**：根据运营反馈不断改进自动化流程
